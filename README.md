## 📖 Project Description

**Amani (أماني)** is a full-stack digital safety ecosystem designed to protect children from the growing risks of the online world while teaching them, through interactive and age-appropriate learning experiences, how to recognize and respond to digital threats. The platform combines a mobile learning application, a secure incident-reporting channel, and an administrative command center into a single cohesive system built on production-grade engineering practices. **This project was developed as an official submission to the Crown Prince Award**, representing a national-level commitment to child safety, technological excellence, and social responsibility in the digital age.

## 🔎 Project Overview

### What is Amani?

Amani is a bilingual (Arabic/English), child-centered digital safety platform composed of three integrated products: a **React Native mobile application** for children, a **React-based admin dashboard** for guardians and moderators, and a **Node.js/TypeScript backend API** that powers both. Together, they form a closed-loop system: children learn, children report, and trained administrators respond.

### Why was it built?

As children spend increasing amounts of time on connected devices, the gap between their digital exposure and their digital literacy continues to widen. Amani was conceived to close that gap by transforming child online-safety education from a passive, one-time lecture into an ongoing, interactive, and rewarding experience — while simultaneously giving children a safe, anonymous, and trustworthy channel to report threats the moment they occur.

### What problem does it solve?

Amani solves the dual problem of **prevention** and **response**. On the prevention side, it teaches children — through stages, lessons, and quizzes — how to identify manipulation, grooming, phishing, and other online dangers. On the response side, it provides a structured reporting pipeline so that when something does go wrong, a trained adult is notified and can act quickly.

### Who are the users?

| User Type | Role |
|---|---|
| 👧 **Children** | Primary learners and reporters, interacting through the mobile app |
| 🧑‍💼 **Administrators / Moderators** | Review reports, manage content, and monitor platform health via the dashboard |
| 👨‍👩‍👧 **Guardians (indirect)** | Benefit from a safer digital environment for their children |

### Why is it important?

Digital safety education at scale is one of the most pressing needs of modern childhood development. Amani directly addresses a Crown Prince Award pillar of innovation for social good by pairing measurable educational outcomes (stage completion, quiz scores, progress tracking) with a real, actionable safety net (the reporting and admin response system).

## ⚠️ Problem Statement

Children today face a wide and evolving spectrum of digital risks, including but not limited to:

- 🎭 **Online grooming** — predators posing as peers to build trust with children
- 📸 **Requests for personal photos or information** — often disguised as games or challenges
- 🔗 **Malicious and phishing links** — shared through chats, games, or social platforms
- 🕵️ **Identity theft and impersonation** — misuse of a child's personal data or images
- 💬 **Cyberbullying and online harassment** — repeated targeting through messages or public posts
- 🚫 **Lack of reporting channels** — most children do not know where or how to report an incident, and many stay silent out of fear or embarrassment

Traditional digital-safety education (posters, one-off school sessions, parental lectures) fails to engage children consistently, and most platforms lack a **child-friendly, judgment-free reporting mechanism**. This leaves a dangerous gap between exposure and protection.

<br>

---

## 💡 Proposed Solution

Amani addresses these risks through a three-layer solution:

1. **Educate** — Gamified, stage-based learning modules teach children to recognize red flags in realistic, age-appropriate scenarios, reinforced through quizzes and rewards that encourage continued engagement.
2. **Empower** — A dedicated, low-friction **Reporting System** lets children flag a concerning experience in a few taps, with pre-categorized report types so they never have to struggle to explain what happened.
3. **Enforce** — A secure, role-based **Admin Dashboard** ensures every report reaches a trained administrator, who can triage, respond, and track resolution — closing the loop between a child's report and real-world action.

## 🚨 Reporting System

The reporting flow is the safety backbone of Amani. Children can submit a report under one of several pre-defined categories, removing the burden of finding the "right words" during a distressing moment.

| Report Type | Description |
|---|---|
| 🗣️ **Someone threatened me** | A user has made threatening statements toward the child |
| 📸 **Someone asked for my photo** | A user has requested personal photos or media |
| 🔗 **Suspicious link** | The child received a link that appears malicious or deceptive |
| 🆔 **Identity theft** | The child's identity, photos, or information have been misused |
| 💬 **Online harassment** | Repeated, targeted bullying or harassment behavior |
| ⚠️ **Other / Uncategorized** | A flexible category for incidents that don't fit predefined types |

Every submitted report is timestamped, categorized, optionally anonymized, and routed directly into the **Admin Dashboard** queue for review.

<br>

## 🖥️ Admin Dashboard

The Admin Dashboard is the operational nerve center of Amani, giving trained administrators full visibility and control over platform safety.

- 📋 **Report Management** — View, triage, and process incoming reports in a unified queue
- 🔍 **Filtering** — Filter reports by type, status, date, or severity to prioritize response
- ✉️ **Replying** — Respond directly to reports where applicable, maintaining a communication trail
- 🔄 **Status Updates** — Move reports through defined states (e.g., New → In Review → Resolved)
- 🔐 **Security** — Role-restricted access ensures only authorized administrators can view sensitive report data
- 📊 **Statistics** — Visual analytics on report volume, categories, response times, and learning engagement across the platform

## 🔐 Environment Variables

Create a `.env` file inside the `backend/` directory using the template below:

```env
PORT=4000

DATABASE_URL=

JWT_SECRET=

JWT_EXPIRES_IN=

ADMIN_EMAIL=admin@amani.com

ADMIN_PASSWORD=123456
```

| Variable | Description |
|---|---|
| `PORT` | The port on which the backend Express server listens |
| `DATABASE_URL` | PostgreSQL connection string used by Prisma to connect to the database |
| `JWT_SECRET` | Secret key used to sign and verify JWT authentication tokens |
| `JWT_EXPIRES_IN` | Expiration duration for issued JWT tokens (e.g. `7d`, `24h`) |
| `ADMIN_EMAIL` | Default administrator email seeded on first setup |
| `ADMIN_PASSWORD` | Default administrator password seeded on first setup — **must be changed in production** |

> ⚠️ **Security Note:** Never commit a populated `.env` file to version control. Always keep credentials out of source control and rotate default admin credentials before any production deployment.

## 🗄️ Database

Amani uses **Prisma** as its ORM layer on top of **PostgreSQL**, providing full type-safety between the database schema and the application code.

```bash
# Apply pending migrations
npx prisma migrate dev

# Generate the Prisma client after schema changes
npx prisma generate

# Push the current schema to the database without generating a migration
npx prisma db push

# Open Prisma Studio — a visual database browser
npx prisma studio

# Seed the database with initial data (admin user, default content, etc.)
npx prisma db seed
```

| Command | Purpose |
|---|---|
| **Migrate** | Creates and applies versioned SQL migrations based on schema changes |
| **Generate** | Regenerates the type-safe Prisma client used throughout the backend |
| **Push** | Syncs the schema directly to the database — useful for rapid prototyping |
| **Studio** | Launches a local GUI to inspect and edit database records |
| **Seed** | Populates the database with default/admin data for local development |

## 🔑 Authentication

Amani uses **JSON Web Tokens (JWT)** for stateless authentication across both the dashboard and mobile clients.

1. A user (admin or child account) authenticates via the `/auth/login` endpoint.
2. On success, the server issues a signed JWT containing the user's identity and role.
3. The client stores the token and attaches it as a `Bearer` token on subsequent requests.
4. Protected routes validate the token's signature and expiration before granting access.
5. Role claims embedded in the token drive **role-based authorization** across the API.

## 🛡️ Security

Amani follows defense-in-depth principles across the backend:

| Layer | Implementation |
|---|---|
| 🪖 **Helmet** | Sets secure HTTP headers to mitigate common web vulnerabilities |
| 🔑 **JWT** | Stateless, signed tokens for authentication and authorization |
| 🔒 **bcrypt** | One-way hashing for all stored user passwords |
| ✅ **Validation** | Zod schemas validate and sanitize all incoming request payloads |
| 🌐 **CORS** | Explicit cross-origin policies restrict which clients may call the API |
| 🗝️ **Environment Variables** | All secrets and credentials are externalized from source code |
